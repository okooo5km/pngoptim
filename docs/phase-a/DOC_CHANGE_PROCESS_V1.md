# Doc Change Process v1

> 生效日期：2026-03-05  
> 关联阶段：A1（方案冻结）  
> 关联条款：`RECONSTRUCTION_MASTER_PLAN.md` 0、10

## 1. 原则

1. 文档先于实现：涉及行为、指标、门禁的改动，必须先更新文档再改代码。
2. 任务可追溯：每个实现任务必须关联阶段与 DoD 条款编号。
3. 证据驱动：每次提交必须附报告或测试证据路径。

## 2. 触发条件

满足任一条件必须先改文档：

1. 参数语义变化（CLI 行为、默认值、退出码）。
2. 验收阈值变化（体积/质量/性能门禁）。
3. 评测方法变化（数据集、参数矩阵、报告结构）。
4. 阶段目标、出口条件或风险豁免标准变化。

## 3. 执行流程

1. 在同一 PR 中先提交文档变更，再提交实现变更。
2. PR 描述中必须填写：
   - 阶段编号（A/B/C...）
   - DoD 编号（如 `DOD-05`）
   - 证据路径（如 `reports/...`）
3. 若仅文档变更，也要说明对后续实现的约束影响。

## 4. 审核清单

1. 约束是否可执行（不是模糊描述）。
2. 阈值是否可测量（有指标定义与口径）。
3. 是否给出回滚路径（特别是优化策略）。
4. 是否同步更新 `AGENTS.md` 阶段状态与最近更新。

## 5. 违规处理

1. 缺失 DoD/证据路径：PR 不可合并。
2. 先改代码后补文档：标记为流程违规，需补齐并复审。
3. 出现无指标优化：退回并要求提供基线对比。
