## 1. 背景与核心决策

### 1.1 已确认决策

1. 使用 **Rust** 作为唯一主开发语言。
2. 采用 **一体化单仓库**（不拆分“引擎库 + CLI 两仓库”）。
3. 研发路径：**先复刻（对标等价）→ 再优化（超越）**。
4. 规避商业授权依赖，优先可开源、可复用方案。
5. 开发模式：AI 编码为主，人类负责方向与验收。

### 1.2 关键事实

- 对标项目优势不仅在算法，也在工程落地（I/O、色彩管理、元数据、错误语义、CLI 行为）。
- 现有公开测试不足以支撑“同等级复刻”，必须自建评测体系。
- “完全复刻”定义为统计意义等价（体积/质量/性能），**不是** bit-exact。

## 2. 目标与非目标

### 2.1 目标（Must）

1. 在统一工具中实现同等级 PNG 量化压缩能力。
2. 体积、视觉质量、性能达到或超过对标基线。
3. CLI 核心参数与语义高兼容。
4. 跨平台稳定（macOS / Linux / Windows）。
5. 建立自动化评测体系与可复现实验报告。
6. 从架构层预留优化通道（SIMD、多线程、策略自适应）。

### 2.2 非目标（Not Now）

1. 不追求逐字节输出一致（bit-exact）。
2. 不优先开发 GUI。
3. 不扩展到 PNG 之外格式。
4. 不在 v1 阶段引入复杂 ML 压缩策略。

## 4. 复刻成功的可验收标准（DoD）

> 所有条款必须可自动化验证。

### 4.1 功能与行为

**DOD-01** 参数兼容覆盖率 ≥ 95%  
核心参数：`--quality --speed --floyd/--nofs --strip --skip-if-larger --output --ext --posterize`

**DOD-02** 退出码语义一致  
至少覆盖：成功、参数错误、质量不足、体积不降、读写失败

**DOD-03** 输入输出流程一致  
支持：文件输入、stdin/stdout、单文件与批处理

**DOD-04** 元数据策略稳定  
`--strip` 与保留模式行为可预测且不损坏文件

### 4.2 压缩效果与视觉质量

**DOD-05** 体积等价或更优  
在标准测试集上，平均输出体积不劣于基线阈值（初始阈值：+1% 内；目标：≤0%）

**DOD-06** 视觉质量达标  
以 SSIM/Butteraugli/PSNR（主+辅）进行门禁，不允许出现明显视觉退化样本

**DOD-07** 专项场景通过  
低色数、半透明边缘、UI 图标、渐变图不得显著劣化

### 4.3 性能与稳定

**DOD-08** 性能不退化  
单线程耗时、多线程吞吐至少一项优于基线，另一项不明显劣化

**DOD-09** 内存峰值受控  
大图处理有明确上限策略，不出现不可控膨胀

**DOD-10** 稳定性  
回归集 + fuzz 场景无崩溃（panic/segfault）

**DOD-11** 跨平台一致性  
三平台统计指标一致（允许微小浮动，不越阈）

**DOD-12** 可复现性  
固定输入+参数+版本，输出统计稳定，CI 可重放

## 5. 一体化架构设计（单仓库）

### 5.1 设计原则

1. 外部一体化产品，内部模块化分层。
2. 算法层与 I/O 层解耦，便于迭代和替换。
3. 每次变更必须可量化收益并可回滚。

### 5.2 逻辑模块

- `cli`：参数解析、错误码映射、任务调度
- `pipeline`：统一流程编排（读取→预处理→量化→重映射→写出）
- `quant_core`：调色板生成、误差模型、抖动/重映射
- `png_io`：PNG 编解码、元数据与颜色配置策略
- `compress_backend`：过滤与压缩后端抽象
- `metrics`：体积、质量、性能指标计算与报告
- `compat`：对标行为语义层
- `tests_harness`：回归、基准、fuzz 驱动

### 5.3 关键架构要求

1. 所有算法策略必须开关化（A/B 比较）。
2. 每个阶段输出结构化指标（JSON/CSV/HTML）。
3. 关键路径可观测（耗时、内存、误差贡献）。

---

## 6. 数据与评测体系设计

### 6.1 数据集分层

1. **功能样本集**：参数行为、退出码、边界输入
2. **质量样本集**：照片、UI、透明、低色、渐变、噪声图
3. **性能样本集**：大图、批量图、极端复杂图
4. **鲁棒性样本集**：损坏文件、异常 chunk、超大尺寸

### 6.2 指标体系

- 体积：输出字节数（均值/中位数/P95）
- 质量：SSIM / Butteraugli / PSNR
- 性能：单图耗时、吞吐、CPU 利用率
- 稳定：失败率、崩溃率、超时率
- 资源：峰值内存、峰值线程数

### 6.3 报告要求

每次关键变更必须生成：

1. 对比基线的差异报告
2. 失败样本清单（含可重放参数）
3. “收益-代价”摘要（质量/体积/性能三角）

## 7. 优化空间充分调研结论（顶层）

> 以下为“应在架构上预埋、后续逐项验证”的方向。

### 7.1 算法层优化方向

1. **颜色距离与误差模型可配置化**  
   - 保留默认策略 + 插拔实验策略
2. **近邻搜索结构可替换**  
   - 允许 VP-tree 之外方案做 A/B
3. **抖动路径精细化**  
   - 局部自适应、并行边界处理、噪声抑制
4. **参数自动选择器**（后期）  
   - 按图像类型自动选择速度/质量/抖动等级

### 7.2 系统层优化方向

1. **数据布局优化（缓存友好）**
2. **SIMD 路径（x86_64 + aarch64）**
3. **并行调度优化（chunk/tile/wavefront）**
4. **大图流式/分块处理降低峰值内存**

### 7.3 编码层优化方向

1. **PNG 过滤策略可实验化**
2. **压缩后端可切换（速度/体积档位）**
3. **元数据处理策略可配置（保留/剥离/白名单）**

### 7.4 观测与调参能力（必须）

1. 每阶段耗时分解
2. 每策略对体积与质量的贡献分解
3. 自动输出“退化根因”提示（哪个模块导致）

## 9. 当前开源方案常见困境与规避策略

### 9.1 困境

1. 协议与商用边界复杂
2. 测试体系偏行为，缺少质量与性能联合门禁
3. 代码历史路径长，局部优化牵一发动全身
4. 可观测性不足，优化收益难归因

### 9.2 规避策略

1. 合规前置（依赖准入 + CI 审查）
2. 评测先行（基线报告先于实现）
3. 模块边界清晰（内部可替换）
4. 指标驱动迭代（每次改动都要有数据证据）
